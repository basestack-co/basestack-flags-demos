# AGENTS.md

## Package manager

- Use `bun` for everything (`bun install`, `bun run <script>`, `bun add`).
- `packageManager` is pinned to `bun@1.3.13`.
- Do NOT use `npm`, `yarn`, or `pnpm` ‚Äî the lockfile is `bun.lock`.

## Dev server

- `bun run dev` starts on **port 3009** (not 3000 ‚Äî the README is stale on this).

## Lint / format / typecheck

- Lint: `bun run lint` (Biome, not ESLint)
- Format: `bun run format` (Biome, not Prettier)
- Check + write: `bun run check`
- There is **no separate typecheck script**. TypeScript is checked implicitly by `next build`.

## Test

- Framework: Vitest + jsdom + Testing Library.
- `bun run test` runs with `--coverage` (100% thresholds on all metrics).
- `bun run test:watch` for watch mode.
- A mock for `next/font/google` is wired via vitest.resolve.alias ‚Üí `test/mocks/next-font-google.ts`.
- Pattern: use `vi.hoisted()` for mock factories in test files before imports (Vitest ESM semantics).
- All `@basestack/flags-react` imports are mocked in tests ‚Äî tests never call the real API.

## Architecture

- **Single Next.js 16 App Router app** ‚Äî no monorepo, no workspaces.
- **`src/libs/feature-flags/config.ts`** ‚Äî the single source of truth for SDK config (env vars) and modal copy config.
- **`src/libs/feature-flags/providers.tsx`** ‚Äî client component wrapping `FlagsProvider` + `FeatureFlagModalsProvider` + theme context. All `"use client"` flag consumers go inside this.
- Flags are **preloaded server-side** in `src/app/layout.tsx` via `fetchFlags()`, passed to the client-side `FlagsProvider` as `initialFlags`.

## Flag consumption patterns

- **Server-side (SSR/RSC):** `fetchFlags(config)` (bulk) or `fetchFlag(slug, config)` (single) from `@basestack/flags-react/server`.
- **Client-side:** `useFlag(slug)` from `@basestack/flags-react/client` ‚Äî returns `{ enabled, payload, isLoading, openFeedbackModal, refresh }`.
- **Client-only guards:** Components that read `useFlag` often use a `mounted` state gate (`useState(false)` + `useEffect(() => setMounted(true), [])`) to avoid SSR hydration mismatches.

## Environment variables

- Required: `NEXT_PUBLIC_BASE_URL`, `NEXT_PUBLIC_PROJECT_KEY`, `NEXT_PUBLIC_ENVIRONMENT_KEY`.
- These are read directly via `process.env.NEXT_PUBLIC_*` in `src/libs/feature-flags/config.ts`.
- `.flagsrc` (gitignored) stores shared CLI config for the `flags-cli` tool.

## Cloudflare / OpenNext deployment

- `bun run preview` / `bun run upload` / `bun run deploy` each run `opennextjs-cloudflare build` first, then the target command.
- `open-next.config.ts` configures the Cloudflare adapter.
- `wrangler.jsonc` defines the worker name (`basestack-flags-demos`), bindings, and assets.
- `cloudflare-env.d.ts` is **auto-generated** by `bun run cf-typegen`. Do not edit manually.
- `initOpenNextCloudflareForDev()` is called in `next.config.ts` to enable local bindings during `next dev`.

## Generated / build artifacts (do not edit)

- `.next/` ‚Äî Next.js build output
- `.open-next/` ‚Äî OpenNext build output
- `cloudflare-env.d.ts` ‚Äî generated by wrangler
- `next-env.d.ts` ‚Äî generated by Next.js
